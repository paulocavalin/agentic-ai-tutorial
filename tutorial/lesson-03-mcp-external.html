What MCP isO que é MCP

MCP stands for Model Context Protocol. It is a standard way for AI applications to talk to tools and data sources. An MCP server exposes capabilities such as search, database access, file inspection, or business workflows. An MCP client connects, discovers those tools, and invokes them through a shared protocol.MCP significa Model Context Protocol. É uma forma padrão de aplicações de AI conversarem com tools e fontes de dados. Um servidor MCP expõe capacidades como busca, acesso a banco, inspeção de arquivos ou workflows de negócio. Um cliente MCP se conecta, descobre essas tools e as invoca por um protocolo compartilhado.

The big win is interchangeability. If two servers speak MCP, your agent code can often swap one for the other without rewriting its tool loop.A grande vantagem é a intercambialidade. Se dois servidores falam MCP, seu código de agent muitas vezes consegue trocar um pelo outro sem reescrever o loop de tools.

Key Insight:Insight Chave: with MCP, tools become a protocol concern instead of an application-specific coding exercise. That reduces glue code and encourages reusable tool ecosystems.com MCP, tools viram uma preocupação de protocolo em vez de um exercício de código específico da aplicação. Isso reduz glue code e incentiva ecossistemas reutilizáveis de tools.

The adapter: converting MCP tools into Agent toolsO adapter: convertendo tools MCP em tools do Agent

The central job of core/mcp_client.py is to translate between two interfaces:O trabalho central de core/mcp_client.py é traduzir entre duas interfaces:

  • MCP side:Lado MCP: async sessions, protocol tool definitions, and protocol tool calls.sessões async, definições de tool do protocolo e chamadas de tool do protocolo.
  • Agent side:Lado Agent: sync Python callables plus OpenAI-style function schemas.callables sync em Python junto com schemas de função no estilo OpenAI.
def _mcp_schema_to_openai(tool: types.Tool) -> Dict[str, Any]:
    return {
        "type": "function",
        "function": {
            "name": tool.name,
            "description": tool.description or "",
            "parameters": tool.inputSchema if tool.inputSchema else {"type": "object", "properties": {}},
        },
    }

This is why the rest of the Agent class does not care that the tool came from MCP. Once converted, it looks like any other function-calling schema from earlier lessons.É por isso que o restante da classe Agent não se importa se a tool veio de MCP. Depois de convertida, ela parece qualquer outro schema de function calling das lições anteriores.

Discovering tools dynamicallyDescobrindo tools dinamicamente

MCPAgentClient.__aenter__() handles the connection lifecycle: open stdio transport, create a session, initialize it, and ask the server what tools exist.MCPAgentClient.__aenter__() cuida do ciclo de vida da conexão: abre o transporte stdio, cria uma sessão, inicializa e pergunta ao servidor quais tools existem.

class MCPAgentClient:
    async def __aenter__(self) -> "MCPAgentClient":
        self._stdio_context = stdio_client(self.server_params)
        self._read, self._write = await self._stdio_context.__aenter__()

        self._session_context = ClientSession(self._read, self._write)
        self._session = await self._session_context.__aenter__()

        await self._session.initialize()
        tools_result = await self._session.list_tools()
        self._tools = tools_result.tools
        return self

The call to session.list_tools() is the protocol-level discovery step. This is why mcp_search_agent.py does not need to hardcode tool schemas the way Lesson 2 did.A chamada para session.list_tools() é a etapa de descoberta no nível do protocolo. É por isso que mcp_search_agent.py não precisa hardcodar schemas de tool como a Lição 2 fazia.

Key Insight:Insight Chave: discovery decouples the agent from server implementation details. The client asks “what can you do?” instead of assuming specific tool definitions ahead of time.discovery desacopla o agent dos detalhes de implementação do servidor. O cliente pergunta “o que você consegue fazer?” em vez de assumir definições específicas de tool com antecedência.

The async ↔ sync bridge problemO problema da ponte async ↔ sync

The MCP SDK is asynchronous, but the course Agent implementation is synchronous. That mismatch is the hardest part of this lesson.O SDK de MCP é assíncrono, mas a implementação de Agent do curso é síncrona. Esse desencontro é a parte mais difícil desta lição.

MCP sideLado MCP

session.call_tool(...) must run on an event loop.session.call_tool(...) precisa rodar em um event loop.

Agent sideLado Agent

tool_registry[name](**kwargs) is expected to be a normal blocking Python function.tool_registry[name](**kwargs) precisa ser uma função Python bloqueante normal.

The solution is split in two directions:A solução se divide em duas direções:

def _make_tool_caller(session, tool_name, loop):
    def call_tool(**kwargs):
        future = asyncio.run_coroutine_threadsafe(
            session.call_tool(tool_name, arguments=kwargs),
            loop,
        )
        return future.result(timeout=120)
    return call_tool

result = await asyncio.to_thread(agent.execute, prompt)

run_coroutine_threadsafe() lets a synchronous function ask the async loop to execute an MCP call. asyncio.to_thread() does the reverse: it runs the blocking agent on a worker thread so the event loop remains alive to serve those tool calls.run_coroutine_threadsafe() deixa uma função síncrona pedir ao loop async para executar uma chamada MCP. asyncio.to_thread() faz o inverso: roda o agent bloqueante em uma thread de trabalho para o event loop continuar vivo e atender essas chamadas de tool.

Building the final tool listMontando a lista final de tools

Once the connection is initialized, get_tools(loop) returns everything the Agent needs: schemas for the model, and a registry of sync wrappers for Python execution.Depois que a conexão é inicializada, get_tools(loop) retorna tudo o que o Agent precisa: schemas para o modelo e um registry de wrappers sync para execução em Python.

def get_tools(self, loop: asyncio.AbstractEventLoop):
    schemas = [_mcp_schema_to_openai(t) for t in self._tools]
    registry = {
        t.name: _make_tool_caller(self._session, t.name, loop)
        for t in self._tools
    }
    return schemas, registry

That tuple shape, (tool_schemas, tool_registry), is intentionally identical to the manually-built agent from earlier lessons. The adapter does the translation once so the Agent can stay unchanged.Esse formato de tupla, (tool_schemas, tool_registry), é propositalmente idêntico ao do agent montado manualmente nas lições anteriores. O adapter faz a tradução uma vez para o Agent permanecer inalterado.

The external server in mcp_search_agent.pyO servidor externo em mcp_search_agent.py

The agent file is surprisingly small because almost all protocol work is hidden in the adapter.O arquivo do agent é surpreendentemente pequeno porque quase todo o trabalho de protocolo fica escondido no adapter.

server_params = StdioServerParameters(command="duckduckgo-mcp-server", args=[])

async with MCPAgentClient(server_params) as mcp:
    tools, registry = mcp.get_tools(loop)
    agent = Agent(client=client, tools=tools, tool_registry=registry)
    result = await asyncio.to_thread(agent.execute, prompt)

duckduckgo-mcp-server is an external program installed on the machine. When launched, it exposes search-related tools over stdio. The agent does not know whether those tools are implemented in Python, Node, or another language; it only sees MCP.duckduckgo-mcp-server é um programa externo instalado na máquina. Quando iniciado, ele expõe tools de busca por stdio. O agent não sabe se essas tools foram implementadas em Python, Node ou outra linguagem; ele só enxerga MCP.
