The Big PictureA Visão Geral

A chatbot takes a prompt and returns text. An agent can also call tools — Python functions that access the real world (APIs, databases, files). The LLM decides which tool to call and what arguments to pass. Um chatbot recebe um prompt e retorna texto. Um agente também pode chamar ferramentas — funções Python que acessam o mundo real (APIs, bancos de dados, arquivos). O LLM decide qual ferramenta chamar e quais argumentos passar.

1 OllamaClient — The LLM TransportOllamaClient — O Transporte do LLM

A thin HTTP wrapper around Ollama's OpenAI-compatible API. It handles the network request and parses the JSON response into typed dataclasses. Um wrapper HTTP fino ao redor da API compatível com OpenAI do Ollama. Ele gerencia a requisição de rede e converte a resposta JSON em dataclasses tipadas. 

# core/client.py

class OllamaClient:
    def __init__(
        self,
        model: str = "gemma4:12b",          # ← default model
        base_url: str = "http://localhost:11434/v1",
        timeout: int = 120,
    ) -> None:
        self.model = model
        self.base_url = base_url.rstrip("/")
        self.timeout = timeout
        self.chat = _Chat(self)              # ← mimics OpenAI structure

The client.chat.completions.create() interface mirrors OpenAI's SDK so you can swap providers later. A interface client.chat.completions.create() espelha o SDK da OpenAI para que você possa trocar de provedor posteriormente.

Key Insight:Insight Chave: The client knows nothing about agents or tools. It just sends messages to an LLM and returns structured responses. This separation means you can test the client independently, or use it for non-agent tasks. O cliente não sabe nada sobre agentes ou ferramentas. Ele apenas envia mensagens a um LLM e retorna respostas estruturadas. Essa separação significa que você pode testar o cliente independentemente ou usá-lo para tarefas sem agentes.

2 Data Models — Typed ResponsesModelos de Dados — Respostas Tipadas

The client returns Python dataclass instances, not raw dicts: O cliente retorna instâncias de dataclass Python, não dicionários brutos: 

# core/models.py

@dataclass
class FunctionCall:
    name: str           # e.g. "get_temperature"
    arguments: str      # JSON string, e.g. '{"city": "Paris"}'

@dataclass
class ToolCall:
    id: str             # unique call ID (for matching results)
    type: str           # always "function"
    function: FunctionCall

@dataclass
class Message:
    role: str           # "assistant"
    content: str        # text response (may be empty if tool_calls present)
    tool_calls: Optional[List[ToolCall]] = None
    reasoning: Optional[str] = None  # chain-of-thought (Gemma supports this)

Having typed models prevents typos in dict key access and gives you IDE autocompletion. Ter modelos tipados evita erros de digitação no acesso a chaves de dicionários e fornece autocomplete no IDE.

3 Agent — The LoopAgent — O Loop

The Agent class is the heart of the framework. It implements a simple but powerful loop: A classe Agent é o coração do framework. Ela implementa um loop simples, mas poderoso: 

# core/agent.py — simplified

class Agent:
    def __init__(self, client, system, tools, tool_registry, max_iterations=8, trace=False):
        self.client = client
        self.tools = tools                # JSON schemas (what the LLM sees)
        self.tool_registry = tool_registry # Python functions (what gets executed)
        self.messages = []
        if system:
            self.messages.append({"role": "system", "content": system})

    def execute(self, message: str) -> str:
        self.messages.append({"role": "user", "content": message})

        for iteration in range(1, self.max_iterations + 1):
            # 1. Send everything to the LLM
            completion = self.client.chat.completions.create(
                messages=self.messages,
                tools=self.tools,
                tool_choice="auto",
            )
            response = completion.choices[0].message

            # 2. Does the LLM want to call a tool?
            if response.tool_calls:
                self.messages.append(response.to_dict())

                for tool_call in response.tool_calls:
                    name = tool_call.function.name
                    args = json.loads(tool_call.function.arguments)
                    fn = self.tool_registry[name]          # ← look up function
                    result = fn(**args)                     # ← execute it

                    self.messages.append({
                        "tool_call_id": tool_call.id,
                        "role": "tool",
                        "name": name,
                        "content": json.dumps(result),
                    })
                continue   # ← go back to step 1 with tool results

            # 3. No tool calls — this is the final answer
            return response.content
Key Insight:Insight Chave: The loop has only two branches: if the LLM returns tool_calls, execute them and loop back. If it returns plain content, that's the final answer. This simple pattern powers every agent in this tutorial. O loop tem apenas dois ramos: se o LLM retorna tool_calls, execute-os e volte ao loop. Se retorna content simples, essa é a resposta final. Esse padrão simples alimenta cada agente neste tutorial.

Building the Weather AgentConstruindo o Agente de Clima

Now let's see how examples/weather_agent.py uses these three components.Agora vamos ver como examples/weather_agent.py usa esses três componentes.

Step 1: Define a tool functionPasso 1: Definir uma função de ferramenta

A tool is just a regular Python function. It receives typed parameters and returns data: Uma ferramenta é apenas uma função Python comum. Ela recebe parâmetros tipados e retorna dados: 

def get_temperature(city: str) -> str:
    """Get the current weather in a given city."""
    if city.lower() == "san francisco":
        return "72"
    if city.lower() == "paris":
        return "75"
    return "70"

This is a mock — in production you'd call a weather API. The agent doesn't know or care whether the function is real or simulated. Isso é um mock — em produção você chamaria uma API de clima. O agente não sabe nem se importa se a função é real ou simulada.

Step 2: Describe the tool with a JSON schemaPasso 2: Descrever a ferramenta com um esquema JSON

The LLM needs a machine-readable description of what the tool does and what parameters it expects. This uses the OpenAI function-calling schema format: O LLM precisa de uma descrição legível por máquina do que a ferramenta faz e quais parâmetros ela espera. Isso usa o formato de esquema de chamada de função da OpenAI: 

get_temperature_tool_schema = {
    "type": "function",
    "function": {
        "name": "get_temperature",           # must match the function name
        "description": "Get the current temperature in a given city.",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "The city to get the temperature for.",
                }
            },
            "required": ["city"],
        },
    },
}
Key Insight:Insight Chave: The schema's description fields are critically important. They are the only way the LLM knows what the tool does. A vague description leads to wrong tool selections; a precise one leads to accurate calls. Os campos description do esquema são criticamente importantes. Eles são a única forma que o LLM tem de saber o que a ferramenta faz. Uma descrição vaga leva a seleções erradas de ferramentas; uma precisa leva a chamadas corretas.

Step 3: Wire everything togetherPasso 3: Conectar tudo

# Create the LLM client
client = OllamaClient(model="gemma4:12b")

# Create the agent with our tool
agent = Agent(
    client=client,
    system="You are a helpful assistant that can answer questions using the provided tools.",
    tools=[get_temperature_tool_schema],         # what the LLM sees
    tool_registry={"get_temperature": get_temperature},  # what gets called
    trace=True,                                  # show the loop internals
)

# Run it
response = agent.execute("What's the weather in Paris?")

The tool_registry is a simple dictionary mapping tool names to Python callables. The key must match the "name" field in the schema — that's how the Agent knows which function to call when the LLM says "call get_temperature". O tool_registry é um dicionário simples que mapeia nomes de ferramentas para callables Python. A chave deve corresponder ao campo "name" no esquema — é assim que o Agent sabe qual função chamar quando o LLM diz "chame get_temperature".

What Happens at RuntimeO Que Acontece em Tempo de Execução

Here's the exact sequence when you run the weather agent with --trace:Aqui está a sequência exata ao executar o agente de clima com --trace:

→ Send to LLM: [system message, user: "What's the weather in Paris?"]→ Envia ao LLM: [mensagem do sistema, usuário: "What's the weather in Paris?"]

→ Send to LLM: [system, user, assistant+tool_call, tool_result: "75"]→ Envia ao LLM: [sistema, usuário, assistente+chamada_ferramenta, resultado: "75"]
← LLM returns: content: "The temperature in Paris is 75°F"← LLM retorna: content: "The temperature in Paris is 75°F"

⚠️ Note:⚠️ Nota: The max_iterations parameter (default: 8) prevents infinite loops. If the LLM keeps calling tools without producing a final answer, the agent returns a fallback message after 8 iterations. O parâmetro max_iterations (padrão: 8) previne loops infinitos. Se o LLM continuar chamando ferramentas sem produzir uma resposta final, o agente retorna uma mensagem de fallback após 8 iterações.

Understanding Trace OutputEntendendo a Saída de Rastreamento

The --trace flag enables _trace_print(), which logs every step of the loop. This is invaluable for debugging: A flag --trace habilita _trace_print(), que registra cada etapa do loop. Isso é inestimável para depuração: 

[TRACE] user
{"content": "What's the weather in Paris?"}

[TRACE] iteration
{"index": 1}

[TRACE] assistant_raw
{"role": "assistant", "content": "", "tool_calls": [...]} 

[TRACE] tool_execution
{"name": "get_temperature", "args": {"city": "Paris"}, "output": "75"}

[TRACE] iteration
{"index": 2}

[TRACE] final_response
"The temperature in Paris is 75°F"

Always use --trace when developing a new agent. It shows exactly what the LLM decided and what tools returned. Sempre use --trace ao desenvolver um novo agente. Isso mostra exatamente o que o LLM decidiu e o que as ferramentas retornaram.
