Implementing Real ToolsImplementando Ferramentas Reais

1 web_search — DuckDuckGo Integrationweb_search — Integração com DuckDuckGo

from ddgs import DDGS

def web_search(query: str, max_results: int = 5) -> List[Dict[str, str]]:
    limit = max(1, min(int(max_results), 10))   # clamp to [1, 10]
    rows = list(DDGS().text(query, max_results=limit))
    results = []
    for row in rows:
        title = row.get("title") or "Untitled"
        url = row.get("href") or row.get("url") or ""
        snippet = row.get("body") or row.get("snippet") or ""
        if not url:
            continue
        results.append({"title": str(title), "url": str(url), "snippet": str(snippet)})
    return results

Uses the ddgs library (DuckDuckGo Search). The function returns a list of dicts — the Agent loop serializes it to JSON before sending it back to the LLM. Usa a biblioteca ddgs (DuckDuckGo Search). A função retorna uma lista de dicionários — o loop do Agent serializa para JSON antes de enviar de volta ao LLM.

⚠️ Note:⚠️ Nota: Tool functions must return JSON-serializable data. The Agent's execute() calls json.dumps(result) on the output. If your function returns a custom class, it will fail. Funções de ferramenta devem retornar dados serializáveis em JSON. O execute() do Agent chama json.dumps(result) na saída. Se sua função retornar uma classe personalizada, falhará.

2 web_fetch — Page Content Retrievalweb_fetch — Recuperação de Conteúdo de Página

def web_fetch(url: str) -> Dict[str, Any]:
    headers = {
        "User-Agent": "SimpleAgent/1.0",
        "Accept": "text/html,application/xhtml+xml,text/plain;q=0.9,*/*;q=0.8",
    }
    response = requests.get(url, timeout=15, headers=headers)
    response.raise_for_status()

    content_type = response.headers.get("content-type", "")
    if "html" in content_type.lower():
        content = _clean_html(response.text)   # strip tags, scripts, styles
    else:
        content = response.text

    return {"url": url, "status_code": response.status_code, "content": content[:3000]}

The [:3000] truncation is important — LLMs have limited context windows, and most of a web page is boilerplate. 3000 chars is usually enough for the main content. O truncamento [:3000] é importante — LLMs têm janelas de contexto limitadas, e a maior parte de uma página web é boilerplate. 3000 caracteres geralmente bastam para o conteúdo principal.

3 Tool SchemasEsquemas de Ferramentas

Each tool needs a JSON schema. Notice how web_search has an optional parameter with constraints: Cada ferramenta precisa de um esquema JSON. Note como web_search tem um parâmetro opcional com restrições: 

web_search_tool_schema = {
    "type": "function",
    "function": {
        "name": "web_search",
        "description": "Search the web and return relevant results with title, URL, and snippet.",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {
                    "type": "string",
                    "description": "Search query to run on the web.",
                },
                "max_results": {
                    "type": "integer",
                    "description": "Maximum number of results to return.",
                    "minimum": 1,
                    "maximum": 10,
                },
            },
            "required": ["query"],    # max_results is optional
        },
    },
}
Key Insight:Insight Chave: Making max_results optional with "minimum"/"maximum" constraints guides the LLM to use reasonable values. The LLM reads these constraints from the schema. You can also clamp values in the function body as a safety net (as we do with min(int(max_results), 10)). Tornar max_results opcional com restrições "minimum"/"maximum" orienta o LLM a usar valores razoáveis. O LLM lê essas restrições do esquema. Você também pode limitar valores no corpo da função como rede de segurança (como fazemos com min(int(max_results), 10)).

The System Prompt: Guiding Multi-Step BehaviorO Prompt de Sistema: Guiando Comportamentos em Múltiplos Passos

In Lesson 1, our system prompt was one line. For a multi-tool agent, the system prompt becomes a behavioral specification: Na Lição 1, nosso prompt de sistema tinha uma linha. Para um agente com múltiplas ferramentas, o prompt de sistema se torna uma especificação comportamental: 

SYSTEM_PROMPT = (
    "Voce e um assistente de pesquisa especializado. Seu objetivo e responder "
    "perguntas complexas com base em evidencias atuais encontradas na web.

"
    "Ferramentas disponiveis:
"
    "- web_search(query, max_results): Pesquisa na web...
"
    "- web_fetch(url): Recupera o conteudo completo de uma pagina web.

"
    "Instrucoes de comportamento:
"
    "1. Antes de pesquisar, pense cuidadosamente em quais queries vao "
    "   encontrar as informacoes mais relevantes.
"
    "2. Faca entre 2 e 4 buscas - nao mais que isso.
"
    "3. Se um resultado parecer incompleto, use web_fetch para mais detalhes.
"
    "4. Na resposta final, sempre cite as fontes utilizadas.
"
    "5. Seja objetivo e estruturado. Use markdown.
"
    "6. Se nao encontrar informacao suficiente, diga isso claramente.
"
    "7. Antes de cada tool call, escreva um plano curto (1-2 frases) "
    "   explicando o proximo passo."
)

The prompt is in Portuguese — this is an educational repo for Brazilian students. The instructions work the same way in any language. O prompt está em português — este é um repositório educacional para estudantes brasileiros. As instruções funcionam da mesma forma em qualquer idioma.

Let's break down what each instruction does:Vamos detalhar o que cada instrução faz:

RuleRegra Why it mattersPor que importa
"2 to 4 searches""2 a 4 buscas" Prevents the agent from burning all 8 iterations on searchesEvita que o agente gaste todas as 8 iterações em buscas
"Use web_fetch for more details""Use web_fetch para mais detalhes" Teaches the model to compose tools (search first, then fetch)Ensina o modelo a combinar ferramentas (buscar primeiro, depois obter)
"Always cite sources""Sempre citar fontes" Forces the model to ground answers in evidenceForça o modelo a fundamentar respostas em evidências
"Write a plan before each tool call""Escrever um plano antes de cada chamada de ferramenta" Makes the agent's reasoning visible (great for debugging)Torna o raciocínio do agente visível (ótimo para depuração)
Key Insight:Insight Chave: System prompts are the primary way to control agent behavior. You're not writing code — you're writing rules that the LLM follows. Iteration count limits, tool sequencing, and output formatting are all controlled through the prompt. Prompts de sistema são a principal forma de controlar o comportamento do agente. Você não está escrevendo código — está escrevendo regras que o LLM segue. Limites de iteração, sequenciamento de ferramentas e formatação de saída são todos controlados pelo prompt.

The Planning PatternO Padrão de Planejamento

Rule 7 in our prompt — "write a plan before each tool call" — enables a powerful debugging pattern. When the LLM wants to call a tool, it first writes a brief explanation in the content field, alongside the tool_calls. A regra 7 em nosso prompt — "escrever um plano antes de cada chamada de ferramenta" — habilita um poderoso padrão de depuração. Quando o LLM quer chamar uma ferramenta, ele primeiro escreve uma breve explicação no campo content, junto com os tool_calls.

Putting It TogetherJuntando Tudo

client = OllamaClient(model="gemma4:12b")

agent = Agent(
    client=client,
    system=SYSTEM_PROMPT,
    tools=[web_search_tool_schema, web_fetch_tool_schema],     # 2 tools
    tool_registry={"web_search": web_search, "web_fetch": web_fetch},
    trace=True,
)

response = agent.execute("What are the latest trends in agentic AI for 2025?")
