The approval checkpoint UIA UI do checkpoint de aprovação

_format_checkpoint() builds the terminal panel shown to the operator. It includes the tool name, exact JSON arguments, and the decision menu._format_checkpoint() monta o painel de terminal mostrado ao operador. Ele inclui o nome da tool, os argumentos JSON exatos e o menu de decisão.

def _format_checkpoint(tool_name: str, args: Dict[str, Any]) -> str:
    lines = [
        "╔══════════════════════════════════════════════════════════════╗",
        "║       ⚠  APPROVAL REQUIRED — DESTRUCTIVE ACTION  ⚠         ║",
        f"  Tool      : {tool_name}",
        f"  Arguments : {json.dumps(args, ensure_ascii=False, indent=4)}",
        "  [A] Approve   [N] Deny   [D] Tool details",
    ]
    return "\n".join(lines)

This is not just cosmetic. Good HITL checkpoints give the human enough context to make a decision quickly. “Approve delete?” is weak. “Approve deleting these exact eight files from this exact bucket?” is actionable.Isto não é só cosmético. Bons checkpoints de HITL dão contexto suficiente para o humano decidir rápido. “Aprovar delete?” é fraco. “Aprovar a exclusão destes oito arquivos exatos deste bucket exato?” é acionável.

Key Insight:Insight Chave: a checkpoint is part audit log, part UX. The better the summary, the less likely the human is to approve blindly.um checkpoint é parte audit log, parte UX. Quanto melhor o resumo, menor a chance de o humano aprovar no automático.

The interactive gate: _ask_approval()O gate interativo: _ask_approval()

The gate prints the checkpoint, optionally short-circuits for demo mode, and then loops on terminal input until the operator approves, rejects, or asks for details.O gate imprime o checkpoint, opcionalmente encurta o caminho no modo demo e depois entra em loop no terminal até o operador aprovar, rejeitar ou pedir detalhes.

def _ask_approval(tool_name: str, args: Dict[str, Any], auto_approve: bool) -> Tuple[bool, str]:
    print(_format_checkpoint(tool_name, args))

    if auto_approve:
        print("[AUTO-APPROVE] Automatic approval enabled.")
        return True, "auto-approved"

    while True:
        choice = input("  Your decision [a/N]: ").strip().lower()
        if choice in ("a", "s", "y", "sim", "yes"):
            return True, ""
        elif choice in ("n", "no", "nao", "não", ""):
            reason = input("  Motivo (opcional): ").strip()
            return False, reason or "rejected by operator"
        elif choice == "d":
            # show tool documentation

Second, the d path provides tool details without leaving the flow.Há duas vantagens sutis aqui. Primeiro, negativas retornam uma string de motivo, o que ajuda o agent a responder com inteligência. Segundo, o caminho d mostra detalhes da tool sem sair do fluxo.

Wrapping the registry with approval logicEnvolvendo o registry com lógica de aprovação

The most reusable function is make_guarded_registry(). It walks the base registry and only wraps tools listed in _DESTRUCTIVE_TOOLS.A função mais reutilizável é make_guarded_registry(). Ela percorre o registry base e só envolve as tools listadas em _DESTRUCTIVE_TOOLS.

def make_guarded_registry(base_registry: Dict, auto_approve: bool, trace: bool) -> Dict:
    guarded = {}
    for name, fn in base_registry.items():
        if name in _DESTRUCTIVE_TOOLS:
            def _make_wrapper(fn_=fn, name_=name):
                def wrapper(**kwargs):
                    approved, reason = _ask_approval(name_, kwargs, auto_approve)
                    if not approved:
                        return {
                            "status": "denied",
                            "tool": name_,
                            "reason": reason,
                            "note": "O agente deve propor uma alternativa ou encerrar a tarefa.",
                        }
                    return fn_(**kwargs)
                return wrapper
            guarded[name] = _make_wrapper()
        else:
            guarded[name] = fn
    return guarded

Notice that the tool schema does not change. The LLM still sees the same tool names and parameters. Only the runtime behavior changes, which keeps the agent prompt simple and the safety logic centralized.Repare que o schema da tool não muda. O LLM continua vendo os mesmos nomes e parâmetros. Só o comportamento em runtime muda, o que mantém o prompt do agent simples e a lógica de segurança centralizada.

How the agent adapts when an operation is deniedComo o agent se adapta quando uma operação é negada

When a destructive tool is rejected, the wrapper returns a structured denial payload instead of throwing an exception. That is important: the agent can read the denial and pivot.Quando uma tool destrutiva é rejeitada, o wrapper retorna um payload estruturado de negação em vez de lançar uma exceção. Isso é importante: o agent pode ler a negativa e pivotar.

{
  "status": "denied",
  "tool": "delete_files",
  "reason": "backup needed for audit",
  "note": "O agente deve propor uma alternativa ou encerrar a tarefa."
}

Because the tool result is still valid JSON, the model can answer with alternatives such as archiving to Glacier, deleting only a subset, or stopping entirely. This is a graceful failure mode, not a crash.Como o resultado da tool continua sendo JSON válido, o modelo pode responder com alternativas como arquivar no Glacier, apagar só um subconjunto ou parar de vez. Isso é um modo de falha elegante, não um crash.

If approvedSe for aprovado

The wrapper calls the real function and the agent reports concrete results such as count, size, and status.O wrapper chama a função real e o agent relata resultados concretos, como quantidade, tamanho e status.

If deniedSe for negado

The wrapper returns context the model can use to suggest safer next steps.O wrapper retorna contexto que o modelo pode usar para sugerir próximos passos mais seguros.

In real production systems, auto-approve should be tightly controlled because it effectively changes the trust policy from “human confirms” to “runtime executes.” For teaching, though, it helps you show the full loop without typing into the terminal.Em sistemas reais de produção, auto-approve deve ser controlado de perto porque ele muda a política de confiança de “o humano confirma” para “o runtime executa”. Para ensino, porém, ele ajuda a mostrar o loop completo sem precisar digitar no terminal.

Key Insight:Insight Chave: HITL is a policy choice. The same wrapper can require manual approval in production and automatic approval in a test harness.HITL é uma escolha de política. O mesmo wrapper pode exigir aprovação manual em produção e aprovação automática em um test harness.
